Common use cases¶

There and back again (aka Bilbo-testing)¶

One of the most common use cases in my personal experience applying one function after the other results in the start state:

  • Serializing an object and deserializing the output should result in the original object, e.g. object -> json -> object
  • Inserting a row into a database and deleting it should result in the original state of the database.
  • Adding a number and subtracting the same number should result in the original number.

Crucially, it doesn't matter what the input here is to the test, these properties should hold true for any and all inputs. No matter what row you insert, deleting it should result in exactly the same start state!

Property basted testing Scala¶

Property based testing libraries exist in many language, the most commonly used one in Scala is ScalaCheck, which can be used with a range of general testing frameworks such as ScalaTest. I will illustrate some of the use cases listed above, but won't go into too much detail of implementations, as this is not supposed to be a ScalaCheck tutorial. Hopefully, it will illustrate what a property-based test looks like and give some inspiration.

Generators¶

ScalaCheck supports a variety of generators for base types, but can easily be extended to support any custom data type. For our example, let's say we have a simple case class for storing information about a customer:

In [2]:
case class Customer(
    name: String,
    address: String,
    age: Int
)
Out[2]:
defined class Customer

We know want to write a generator for this case class before we can use it in any property-based tests. This is very straight-forward by implementing generators for String and Int first. For integers, this is super easy, while for strings, we need to specify how long we want our string to be:

In [3]:
import org.scalacheck.Gen

val intGen: Gen[Int] = Gen.choose(Integer.MIN_VALUE, Integer.MAX_VALUE)
val stringGen: Gen[String] = for {
      // String shoud have length between 1 and 20
      n <- Gen.choose(min = 1, max = 20)
     // Choosing any alpha-numeric char n times and concat
      string <- Gen.listOfN(n = n, g = Gen.alphaNumChar).map(_.mkString)
    } yield string

Now that we have the basic generators defined, we can define one for our case class Customer:

In [6]:
val customerGen: Gen[Customer] = for {
      name <- stringGen
      address <- stringGen
      age <- intGen
    } yield {
      Customer(
        name = name,
        address = address,
        age = age
      )
    }
(1 to 3).map(_ => customerGen.sample)

Perfect - with minimal effort, we create a generator for our case class. Of course, this also work with nested case class, or any other data structure you have. Typically, I define these generators once and stick them into a shareable trait or object for all tests. This makes the effort to introduce new tests minimal.

Testing serialization¶

Let's test if we can serialize and deserialize Customer instances correctly. To serialize into json, we'll use circe:

In [7]:
import io.circe.generic.auto._
import io.circe.parser
import io.circe.syntax._

val exampleCustomer = Customer(
    name = "Fred",
    address = "San Francisco",
    age = 25
    )

val encodedJson = exampleCustomer.asJson // encode as Json
val decodedJson = parser.decode[Customer](encodedJson.toString)
Out[7]:
import io.circe.generic.auto._

import io.circe.parser

import io.circe.syntax._


exampleCustomer: Customer = Customer("Fred", "San Francisco", 25)
encodedJson: io.circe.Json = JObject(
  object[name -> "Fred",address -> "San Francisco",age -> 25]
)
decodedJson: Either[io.circe.Error, Customer] = Right(
  Customer("Fred", "San Francisco", 25)
)

Now, if you wanted to unit test that the decodedJson matches your original input exampleCustomer, you can obviously write a quick test that asserts that they are equal: decodedJson == Right(exampleCustomer). But - you only tested one input! What if this doesn't work with really long strings, or really large integers? Writing a unit test for each of these edge cases would take a long time, so let's utilize our generators from earlier and specify a property:

In [8]:
import org.scalacheck.Prop._
import org.scalacheck.Prop

val serializingProp: Prop = forAll(customerGen) { customer =>
  val json = customer.asJson.toString
  parser.decode[Customer](json) == Right(customer)
}
Out[8]:
import org.scalacheck.Prop._

import org.scalacheck.Prop


serializingProp: Prop = Prop

You can see that the type of is Prop - a property. We only specified something that should hold true, but haven't tested it yet. By calling check on the property, we actually test if the property holds true:

In [9]:
serializingProp.check

+ OK, passed 100 tests.

So with minimal effort, we ran the same test with 100 inputs rather than just one hand crafted one! (By default, ScalaCheck uses 100 inputs, but you can change this to whatever you like.)

For fun, let's specify a property that doesn't hold true and see what happens:

In [10]:
val failingProp: Prop = forAllNoShrink(stringGen) { s => s == s.reverse }
Out[10]:
failingProp: Prop = Prop

(Don't worry about the forAllNoShrink instead of forAll, but if you're interested in the concept of shrinking, read this blog post)

Now, this property will clearly fail at some point - maybe we get lucky and the generator creates some palindromes by chance, but that's quite unlikely. And indeed, it fails right away and tells what argument made it fail:

In [11]:
failingProp.check()

! Falsified after 0 passed tests.
> ARG_0: "osbqcZjqqaxb"

And that's it: With a few lines of code, we extensively tested JSON serializing and deserializing. Hopefully, this will help you get started with finding your own use cases for property-based testing.

Integration with ScalaTest¶

My preferred way of using ScalaCheck is with ScalaTest, as that is the testing framework I already use for regular unit tests. ScalaTest provides a handy ScalaCheckDrivenPropertyChecks trait that you can mixin for some syntactic sugar, which makes importing functions from ScalaCheck directly unnecessary. Here is an example of usage with FlatSpec:

In [15]:
import org.junit.runner.RunWith
import org.scalatest.FlatSpec
import org.scalatest.Matchers
import org.scalatestplus.junit.JUnitRunner
import org.scalatestplus.scalacheck.ScalaCheckDrivenPropertyChecks

@RunWith(classOf[JUnitRunner])
class CustomerTest extends FlatSpec
    with Matchers with ScalaCheckDrivenPropertyChecks {

  "Customer case class" should "be an serializable to/from JSON" in {
    forAll(customerGen) { exp =>
      val json = exp.asJson.toString
      parser.decode[Customer](json) shouldBe Right(exp)
    }
  }
}
Out[15]:

import org.scalatest.FlatSpec

import org.scalatest.Matchers

After that, you can run it right in IntelliJ or through whatever IDE you use, as well as through any build tools like SBT, Gradle or Maven.
